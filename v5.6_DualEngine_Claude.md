## v5.5 Audit: Why v5.6 Was Necessary

### Primary Bug: `p_short_max_for_long` Dead Zone

The v5.5 search space used `p_short_max_for_long: [0.55, 0.90]`. Values above
`conf_thresh_short = 0.786` are **inert** — they add zero additional long signals.

**Signal evaluation order (backtest.py):**
```python
short_mask = (p_short >= conf_thresh_short) & (p_short >= p_long)  # evaluated FIRST
long_mask  = (p_long  >= conf_thresh_long)  & (p_short <= p_short_max_for_long)
direction  = where(short_mask, -1, where(long_mask, 1, 0))
```

When `0.786 <= p_short <= p_short_max`:
- If `p_short >= p_long`: `short_mask` fires → direction = -1 (long never activates)
- If `p_long > p_short`: `short_mask` already fails → long fires regardless of `p_short_max`

**Net result:** Any `p_short_max > 0.786` is functionally identical to `0.786`. The v5.5
optimizer spent ~37% of its search budget (0.786→0.90) in this dead zone and converged on
Trial #289 with `p_short_max = 0.869` — a meaningless value. This directly caused the
fold 4 long WR collapse (31.82%).

## v5.6 Design Decisions

### 1. Search Space Corrections

| Parameter           | v5.5 Range    | v5.6 Range    | Rationale                                |
|---------------------|---------------|---------------|------------------------------------------|
| `conf_thresh_long`  | [0.50, 0.70]  | [0.55, 0.70]  | Remove sub-0.55 noise entries            |
| `tp_mult_long`      | [2.0,  5.0]   | [2.0,  3.5]   | Tighter TP → faster closes → more trades |
| `sl_mult_long`      | [0.70, 1.50]  | [0.70, 1.20]  | Tighter SL → better R:R at density      |
| `rsi_1d_long_limit` | [45,   65]    | [45,   65]    | Unchanged                                |
| `p_short_max_for_long` | [0.55, 0.90] | [0.55, 0.75] | Entirely below dead zone (0.786)         |

### 2. Quality Gate Raised

```
Per-fold long WR gate:  0.33 → 0.41
Min total long trades:    45 →   80   (density floor; target 120–180)
```

Folds failing the 41% WR gate score `-10.0` (penalty signal to pruner) rather than
being silently skipped. This forces TPE away from WR-weak regions.

### 3. Pruner: MedianPruner (not HyperbandPruner)

HyperbandPruner aggressively kills trials after early folds. Long Runner quality is
often back-loaded — fold 5 frequently shows the best WR. MedianPruner uses a gentler
comparison against the median of completed fold scores, preventing premature killing.

```python
optuna.pruners.MedianPruner(
    n_startup_trials = 5,
    n_warmup_steps   = 2,   # let 2 folds complete before pruning
    interval_steps   = 1,
)
```

### 4. Objective Function (unchanged from v5.5)

```
Density Score = (WR_long × Count_long × ln(Count_long + 1)) / DD_Impact
```

- `WR_long`: per-fold long win rate (0.0–1.0)
- `Count_long`: number of long trades in fold
- `ln(Count+1)`: log reward — diminishing returns past ~180 trades
- `DD_Impact = max(|max_drawdown|, 0.01)`: penalises drawdown

DD penalty multiplier (0.40×) applies when `|fold_dd| > 12%`.

### 5. Long Runner Carries All v5.5 Code Fixes

The following v5.5 improvements are preserved unchanged:

- `rsi_1d_long_limit` injectable via `params` dict (no hardcoded fallback in hot path)
- `minor_mss_lookback` injectable via `params` dict
- `p_long > p_short` directional gate **removed** from long signal logic — long fires
  purely on `p_long >= conf_thresh_long AND p_short <= p_short_max_for_long`
